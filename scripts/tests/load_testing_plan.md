# План нагрузочного тестирования Nuxt frontend

## Что проверяем

Сценарии ниже составлены по текущему frontend-потоку:

- регистрация и автологин: `frontend/app/components/SignupCard.vue`, `frontend/app/stores/auth.js`
- список событий: `frontend/app/components/EventList.vue`
- экран выбора мест: `frontend/app/pages/seats/[event_id].vue`
- схема мест и действия по месту: `frontend/app/components/SeatsList.vue`
- корзина и подтверждение броней: `frontend/app/components/BookingCart.vue`
- Nuxt proxy до Django: `frontend/server/api/backend/[...path].ts`
- websocket синхронизация схемы: `frontend/app/composables/useSeatmapRealtimeSync.js`

Важно: при каждом изменении места frontend не только делает `POST hold/confirm/release`, но и:

- обновляет `seatmap`
- обновляет список броней пользователя
- получает websocket-событие `seatmap.change`
- заставляет остальных открытых пользователей тоже обновлять `seatmap`, а для обычных пользователей еще и `bookings`

Из-за этого самый тяжелый сценарий для системы не регистрация, а одновременное присутствие пользователей на странице схемы мест.

## Реальный путь пользователя через frontend

### 1. Новый пользователь

1. `GET /signup`
2. `GET /api/backend/accounts/signup-groups/`
3. `POST /api/backend/accounts/signup/`
4. `POST /api/backend/accounts/token/`
5. `GET /api/backend/accounts/me/`
6. `GET /`
7. `GET /api/backend/booking/events/`

### 2. Переход к бронированию

1. `GET /seats/<event_id>`
2. `GET /api/backend/booking/events/<event_id>/`
3. `GET /api/backend/booking/events/<event_id>/seatmap/`
4. `GET /api/backend/booking/bookings/?event_id=<id>&status=held,booked&active_only=true`
5. `WS /ws/booking/events/<event_id>/seatmap/`

### 3. Выбор места

1. `POST /api/backend/booking/hold/`
2. `GET /api/backend/booking/events/<event_id>/seatmap/`
3. `GET /api/backend/booking/bookings/?event_id=<id>&status=held,booked&active_only=true`
4. websocket-событие вызывает обновления у других открытых клиентов

### 4. Подтверждение

1. `POST /api/backend/booking/confirm/`
2. дальнейшие websocket-обновления

## Главные узкие места

1. Все API-вызовы идут через Nuxt proxy, поэтому под нагрузкой нужно мерить и frontend-сервер, и Django.
2. `seatmap` всегда возвращает весь зал, то есть все 600 мест, а не только изменившееся место.
3. На `hold` есть блокировка строки места в БД, поэтому конкуренция за один и тот же ряд или сектор будет чувствоваться сильнее обычного.
4. Один клик по месту может породить волну дополнительных `GET` у других пользователей, которые просто смотрят схему.

## Роли пользователей в тесте

Чтобы тест был похож на реальность, лучше смешивать несколько ролей:

- `Регистрация`: регистрируется, автоматически входит, открывает список событий.
- `Покупатель`: открывает схему, выбирает 1-3 места, подтверждает бронь.
- `Наблюдатель`: открывает схему и держит страницу открытой 2-5 минут, почти не кликает, но держит websocket.
- `Конкурент`: пытается взять место в популярной зоне, где уже есть другие попытки.

## Рекомендуемые сценарии

### Сценарий A. Базовый прогон

- Одновременных пользователей: `10`
- Состав:
- `4` регистрация
- `4` покупатели
- `2` наблюдатели
- Ramp-up: `5 минут`
- Длительность удержания нагрузки: `10 минут`
- Задача: проверить, что весь путь работает без роста ошибок

### Сценарий B. Обычная рабочая нагрузка

- Одновременных пользователей: `25`
- Состав:
- `8` регистрация
- `10` покупатели
- `7` наблюдатели
- Ramp-up: `8 минут`
- Длительность удержания нагрузки: `15 минут`
- Задача: смоделировать спокойный реальный поток, когда люди заходят постепенно

### Сценарий C. Реалистичный пик

- Одновременных пользователей: `50`
- Состав:
- `10` регистрация
- `25` покупатели
- `15` наблюдатели
- Ramp-up: `10 минут`
- Длительность удержания нагрузки: `20 минут`
- Задача: основной сценарий для ваших ожиданий в `200` пользователей за событие

Почему именно этот сценарий основной:

- при аудитории `200` человек часто одновременно активны `25-35%`
- это дает примерно `50-70` одновременных пользователей
- из них часть только смотрит зал, но все равно держит websocket и создает фоновую нагрузку

### Сценарий D. Верхняя граница реалистичного пика

- Одновременных пользователей: `75`
- Состав:
- `10` регистрация
- `40` покупатели
- `25` наблюдатели
- Ramp-up: `12 минут`
- Длительность удержания нагрузки: `20 минут`
- Задача: проверить запас перед пиковым открытием продаж или публикацией ссылки в чате

### Сценарий E. Защитный spike-тест

- Одновременных пользователей: `100`
- Состав:
- `10` регистрация
- `55` покупатели
- `35` наблюдатели
- Ramp-up: `2 минуты`
- Длительность удержания нагрузки: `8 минут`
- Задача: проверить кратковременный всплеск, когда много людей пришли почти одновременно

Этот сценарий уже выше реалистичного ожидания, но полезен как запас прочности.

### Сценарий F. Конфликт за популярные места

- Одновременных пользователей: `30`
- Состав:
- `5` регистрация
- `20` конкуренты
- `5` наблюдатели
- Ramp-up: `3 минуты`
- Длительность удержания нагрузки: `10 минут`
- Поведение:
- `20` пользователей пытаются брать места из одного и того же небольшого диапазона, например `20-30` мест в центре
- при `409 CONFLICT` делают паузу `2-5` секунд и пробуют соседнее место
- Задача: проверить блокировки БД, всплеск конфликтов и каскад websocket-обновлений

## Какой набор сценариев запускать в первую очередь

Если времени мало, запускать в таком порядке:

1. `A` чтобы проверить, что сценарий вообще корректный
2. `C` как главный бизнес-сценарий
3. `D` как проверку запаса
4. `F` как самый рискованный по конкуренции за места
5. `E` как краткий стресс-тест

## Рекомендуемые паузы между действиями

Чтобы тест был похож на живых людей, не запускайте действия без think time.

- после открытия `/signup`: `2-6` секунд
- после логина или автологина: `1-3` секунды
- после открытия списка событий: `2-5` секунд
- на схеме мест до первого клика: `5-20` секунд
- между повторными попытками выбора места: `2-5` секунд
- перед подтверждением брони: `10-30` секунд

## Что считать успешным результатом

Для первых прогонов можно использовать такие ориентиры:

- доля ошибок `5xx`: меньше `1%`
- доля ошибок `4xx` вне конфликтов на место: меньше `1%`
- `409 CONFLICT` в сценарии `F`: допустимы и ожидаемы
- `GET /api/backend/booking/events/<id>/seatmap/` p95: до `800 мс`
- `POST /api/backend/booking/hold/` p95: до `1200 мс`
- `POST /api/backend/booking/confirm/` p95: до `1500 мс`
- websocket reconnect rate: близко к `0`

## Практический вывод для вашей нагрузки

При ожидании `200` пользователей я бы ориентировался на такую целевую проверку:

- обязательный проход: `50` одновременных пользователей
- желательный запас: `75` одновременных пользователей
- короткий стресс-предел: `100` одновременных пользователей

Если система уверенно держит `75` одновременных пользователей в сценарии `D` и не разваливается на `100` в сценарии `E`, то для вашего кейса это уже хороший запас.

## Что дальше автоматизировать

Текущие файлы `scripts/tests/http_load_test.py` и `scripts/tests/websocket_test.py` являются заглушками и не отражают текущий Nuxt-поток.

Следующий шаг:

- сделать один сценарий на `Locust` или `k6` для HTTP-пути через Nuxt `/api/backend/...`
- отдельно сделать сценарий удержания websocket-подключений к `/ws/booking/events/<id>/seatmap/`
- запускать их вместе, потому что основная нагрузка у вас смешанная: HTTP + websocket
